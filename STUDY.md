# One Spec, Five Agents: A Controlled Comparison of AI Coding Systems

*The experiment: November 2025, by Scott McGuire — one fully specified project handed to five AI coding agents, each building it independently. The evaluation: July 2026, after the artifacts resurfaced — performed by a fleet of AI agents under his direction, and checked against code and execution rather than the builds' own claims. A first-draft analysis was itself re-verified by an independent execution pass and corrected where measurement contradicted it (see Method). A companion piece, written by Claude — this repository's [README](README.md) — tells the story of the experiment and its recovery; this document is the comparison itself.*

## The setup

One spec-driven project — a project-aware multi-model chat system (filesystem as source of truth, sandboxed Docker execution, a unified FTS5 search index) — was written up as a vision doc, an architecture doc, a roadmap, and nine numbered feature specs. In November 2025, days after Gemini 3 and Antigravity launched, the *same* pack was handed to five AI coding systems, each dispatched with a single instruction — "Please build this project from beginning to end according to the plan" — and left to build it independently:

| Agent | Underlying model |
|---|---|
| Claude Code | Sonnet 4.5 |
| Factory Droid | Sonnet 4.5 |
| Codex | GPT-5 codex-max |
| Gemini | Gemini |
| Antigravity | Google Antigravity |

One honest framing point up front, because it changes what the comparison measures: **the nine specs don't just describe the features — each ships a near-complete, copy-pasteable reference implementation.** So the task the agents actually faced was not "invent this from scratch." It was closer to "transcribe a large reference faithfully, wire it together into something that runs, and decide where — and whether — to diverge and improve." That makes the interesting signal not *can it code*, but *how far along the arc does it get, what does it do at the points where transcription isn't enough, and does it tell the truth about what it finished.*

## What the experiment shows

Three findings.

**1. The agents complete the documented plan — not the spec inventory.** The nine specs form an arc: 01–04 are pure infrastructure (schema, storage, conversations, Docker), 05–08 assemble the product (adapters, tool loop, search, prompts, UI), and 09 is conversation-context-management (token estimation, summarization, truncation). **Every build nailed the infrastructure. Only one of five attempted spec 09.** The first-draft analysis attributed this to reference thinness — spec 09 as the one place agents had to *write* rather than *transcribe*. Measurement killed that: spec 09's reference is as complete and copy-pasteable as the others (449 fenced code lines, third-smallest of nine, full implementations of the summarizer and endpoints), and Codex's spec-09 code is a near-verbatim transcription of it. The actual discriminator is more interesting: **spec 09 is missing from every artifact that encodes the plan** — the specs/README implementation order (`01→02→04→05→07→08`), its time estimates, and the ROADMAP dependency graph. Four of five agents did what the plan said and stopped where the plan stopped; none of them noticed that the plan covered eight of the nine specs sitting in the same directory. Only Codex reconciled the plan against the spec inventory and built what it found there. That is a real, if different, statement about current coding agents: they are plan-followers, not spec-auditors — hand them an internally inconsistent requirements pack and they inherit its omissions silently. For anyone using agents on real projects, the implication is sharp: the completeness of what you get is bounded by the completeness of your *index*, not your specs.

**2. Same model, different harness, materially different result.** Two of the builds ran on *the same underlying model* — Sonnet 4.5 — under different agent scaffolds: Claude Code and Factory Droid. They landed in different places (75% vs 70%) with *different failure modes*: Claude Code got further, proved it ran, but shipped search inert and contradicted its own progress notes; Droid stopped short on search, left its docs stale, and overclaimed completion. (Instructively, one defect the two builds *share* — consecutive assistant turns left unmerged in multi-model rounds, so other models' replies read as the model's own — which suggests a model-level habit showing through both scaffolds, while everything else about where they landed diverged.) The scaffold — how the agent plans, checks itself, and decides it's done — shaped the outcome as much as the model did. For anyone evaluating "which model codes best," this is the confound: you are never evaluating a model, you are evaluating a model-in-a-harness.

**3. "Looks finished" and "is wired" are different properties, and self-reports run backwards to competence.** The most common failure was not broken code — it was *plausible* code that isn't connected: Claude Code's indexer that nothing calls, Gemini's `dotenv` dependency that nothing invokes (so its own documented setup crashes at boot), Antigravity's route files for a framework it never installed. Each looks complete on a file listing and fails on execution. And the completion claims ran opposite to competence: the most complete build (Codex) made one modest, *accurate* claim; the second most complete (Gemini) claimed nothing; the build that claimed "all 8 steps done" (Droid) had skipped one; the build that ran end-to-end (Claude Code) shipped two mutually contradictory status files. Calibration, not silence, is the competence signal — and if you grade these builds by their own docs, you rank them almost backwards. The only reliable signal was running the code and reading the wiring — which is the whole argument for evaluation over self-attestation.

## Per-agent, briefly

- **Codex (GPT-5 codex-max)** — the most complete and the highest-fidelity: storage layout, FTS5 schema, ephemeral Docker, and the subtle OpenAI-vs-Anthropic system-prompt contract all match the architecture; it's the only build that implemented spec 09, and its search passes end-to-end when executed. Real gaps are small: the Google adapter is a throwing stub, and two of its own test scripts can't run without live API keys (its adapters construct clients at module import).
- **Gemini** — a faithful transcription with the best *judgment at the margins*: it's the only build that caught and fixed bugs present in the reference itself (a UI event-handler bug, a missing `/api/projects` route layer the UI depended on, a circular `require`). It skipped the same plan-omitted spec everyone but Codex skipped, and it left one genuine footgun (a declared-but-unwired `dotenv`: follow its documented setup and the server crashes at startup).
- **Claude Code (Sonnet 4.5)** — the only build with *proof of life from the experiment itself*: a saved gpt-4o conversation turn with real usage tokens, plus a live authenticated API error in its server log. Strong, clean core spine. Undone by an inert search feature and self-contradictory status docs.
- **Factory Droid (Sonnet 4.5)** — an architecture-faithful core with two clean misses (search, context-mgmt), a latent turn-attribution defect in multi-model rounds, stale documentation, and a completion overclaim.
- **Antigravity** — the cleanest *incomplete* build: everything it shipped (the infrastructure layers) is tested and correct, and it made no false claims. It simply stopped where the product starts.

## Method and limits

Each build was assessed against a per-spec acceptance bar extracted from the nine specs (every spec ships a standalone `test-*.js`, which makes completeness objectively checkable). Assessment was code-first and adversarial toward each build's own documentation — including toward the first draft of this study: a second, independent verification pass re-derived every claim, executed all runnable suites (footnote 1), booted the servers, and corrected the first draft where it was wrong (most notably the mechanism of finding 1, which flipped from "couldn't write original logic" to "followed an incomplete plan" when the spec-09 reference was actually measured). What remains undone: driving a live multi-model turn through each build (needs real API keys) and exercising the Docker execution paths. The completion frontier is execution-checked; those two verdicts could still tighten. One publication note: the build artifacts as archived include environment files with real credentials — these must be stripped (and the keys rotated) before any of this is released.

## Provenance

The experiment's subject was not hypothetical. A working version of the same app — built the month before, iterated mostly through a single harness, and in daily use — preceded it; the experiment was a spec-driven, controlled rebuild of that idea. The five builds then sat unexamined for eight months before the evaluation above was performed; how they resurfaced is told in the companion piece. What the comparison offers is the thing an evaluation-minded reader cares about — not a leaderboard number, but *where different coding agents diverge when handed identical, rigorous requirements, and how to tell a finished-looking build from a working one.* The answer, here: agents inherit their plan's omissions without auditing it; the harness matters as much as the model; and an agent's own account of what it finished tracks its calibration, not its competence.
